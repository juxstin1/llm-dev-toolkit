# SPEC-001: CLI Ergonomics And Workflow Shortcuts

> `tk` should keep its stable Unix-shaped command surface while adding a faster,
> more discoverable layer for daily codebase work: visible aliases, shorter
> common flags, natural file inspection, and later workflow commands for search
> context and repo orientation.

## Problem

`tk` already covers the right jobs: list, find, search, inspect, count, hash,
summarize, and expose read-only MCP tools. The friction is that common commands
can feel longer than they need to, and the help/docs surface makes users scan a
large flat command list before discovering the fast path.

The goal is not to replace the existing CLI. The goal is to make the shortest
good command obvious.

## Current State Evidence

| Area | Evidence |
| --- | --- |
| CLI parser | `src/main.rs` defines one top-level `Commands` enum and dispatches all commands from `main()`. |
| Existing aliases | `src/main.rs` has hidden clap aliases for `ls -> l`, `tree -> lt`, and `search -> grep`; `fd` is a separate command variant that dispatches to the same runner as `ff`. |
| Global output flags | `src/main.rs` exposes `--format <text|json>` and `--color <auto|always|never>` as long global flags. |
| Search line numbers | `SearchArgs.line_number` has only `--line-number`; there is no `-n` shorthand. |
| File info | `InfoArgs` requires `-f/--file <path>` for file details; bare `info` shows system overview. |
| Find surface | `ff`, `fd`, `ff-ext`, and `ff-name --glob` split overlapping file discovery workflows. |
| Help surface | `target\release\tk.exe --help` lists many top-level commands but does not show hidden aliases or common recipes. |
| README surface | `README.md` has a flat command table plus examples, but no grouped "daily driver" path before the full reference. |
| MCP surface | `src/mcp.rs` exposes canonical read-only tool names and shells into `tk ... --format json --color never`; MCP should remain stable and minimal. |
| Tests | `tests/cli.rs` already contains CLI smoke tests, alias tests for `grep`, JSON tests, help tests, and MCP integration tests. |

## Goals

- Keep all current canonical commands and flags working.
- Add shortcuts as aliases or flag aliases rather than renames.
- Make the root help output teach the most useful commands quickly.
- Make the README guide users by workflow before showing the full reference.
- Add tests for each public shortcut that changes parsing behavior.
- Keep MCP canonical and deterministic; do not expose human-only aliases as
  extra MCP tool names unless a future MCP spec explicitly asks for that.
- Define later workflow commands in enough detail that they can be built in
  separate slices.

## Non-Goals

- Do not remove `ff`, `fd`, `search`, `tree`, `ltd`, or any current command.
- Do not make broad behavior-changing default changes in this spec, such as
  enabling search line numbers by default.
- Do not move validation to clap `ValueEnum` unless a later spec revisits
  parse-time versus command-time JSON error behavior.
- Do not add config files or user-local defaults in this spec.
- Do not expose mutating commands through MCP.
- Do not require full video rendering or demo changes for CLI ergonomics.

## Proposed Design

### Decisions

This spec accepts these defaults for implementation:

- One-letter aliases `s`, `f`, and `t` are public visible aliases.
- `fd` remains supported as a compatibility command, but docs should present it
  as part of the `ff`/find family rather than as a separate concept.
- `search` line numbers remain opt-in; `-n` is the fast path.
- `rg` is acceptable as a visible alias if help/docs clearly describe substring
  search rather than ripgrep-compatible regex behavior.
- `scan` is human-first in text mode and automation-friendly in JSON mode.
- MCP remains canonical; aliases do not become extra MCP tools.
- Shell completions should be generated by a runtime command, not checked-in
  generated artifacts.

### Phase 1: Additive Fast-Hands Layer

Add concise aliases and short flags while preserving every existing command.

#### Command Alias Contract

| Canonical command | Existing surface | Add or promote | Notes |
| --- | --- | --- | --- |
| `ls` | hidden alias `l` | make `l` visible | Keep `la` and `ll` as shortcut commands because they change flags. |
| `tree` | hidden alias `lt`; separate `ltd` shortcut | make `lt` visible; add `t` | Keep `ltd` for compatibility, but docs should promote `tree -L` / `t -L`. |
| `ff` | `fd` variant; `find` alias on `fd` | add visible aliases `f`, `fd`, `find` to the canonical find surface | Implementation may keep `Fd` as a wrapper initially, but help/docs should present one find concept. |
| `ff-ext` | no alias | add `fe` | Short extension lookup. |
| `ff-name` | no alias | add `fn` | Short name/glob lookup. |
| `search` | hidden alias `grep` | make `grep` visible; add `s`, `rg` | `rg` is an ergonomic alias only; behavior remains substring search, not ripgrep-compatible regex. |
| `preview` | no alias | add `view` | Avoid `bat` unless the project wants to imply bat-compatible behavior. |
| `count` | no alias | add `wc` | Useful because flags already mirror word-count style. |
| `checksum` | no alias | add `hash`, `sum` | Keep algorithm flag unchanged. |
| `largest` | no alias | add `big` | Short daily-driver alias. |
| `recent` | no alias | add `new` | Short daily-driver alias. |

Aliases should be visible in help when they are part of the public contract.
Hidden aliases are allowed only for compatibility or typo handling.

#### Flag Shortcut Contract

| Current flag | Add | Meaning | Notes |
| --- | --- | --- | --- |
| `--format json` | `-j`, `--json` | Emit JSON output. | Equivalent to `--format json`. Mixing `--json` and `--format text` should be rejected or resolved explicitly in implementation. |
| `--color never` | `--no-color` | Disable ANSI output. | Equivalent to `--color never`. Mixing `--no-color` and `--color always` should be rejected or resolved explicitly in implementation. |
| `search --line-number` | `search -n` | Show line numbers. | Do not make line numbers default in this phase. |
| `sort --by <field>` | `sort -k <field>`, optionally `--key <field>` | Sort key. | Keep `--by` working. |
| `stats -t` | `stats -e`, optionally `--ext` | Per-extension breakdown. | Keep `-t/--by-type` working. |
| `clip --allow-file-fallback` | `clip --fallback` | Opt in to persistent fallback storage. | Keep explicit safety wording in long help. |

### Phase 2: Natural File Info

Allow:

```powershell
tk info README.md
```

to mean the same as:

```powershell
tk info -f README.md
```

Bare `tk info` must continue to show system overview. Supplying both positional
path and `-f/--file` should either fail with a clear error or require the values
to match; the preferred design is to reject both to avoid ambiguity.

### Phase 3: Help And README Discoverability

The help and docs should lead with common workflows, not just the full command
inventory.

Root help should include a compact recipe block, for example:

```text
Common:
  tk ll                 long listing with hidden entries
  tk t -L 2             tree, two levels deep
  tk f config -e toml   find config files
  tk s TODO src -n      search with line numbers
  tk stats -e -j        extension stats as JSON
```

README should be reorganized into:

1. Install.
2. First 60 seconds.
3. Command map by job.
4. Full command reference.
5. JSON and MCP reference.
6. Development.

Docs should also refresh stale planning/proof references so completed build
slices are not described as current bugs.

### Phase 4: Search-To-Context Workflow

Add a read-only command that jumps from a `path:line` search result into useful
context.

Preferred command names:

- canonical: `show`
- alias: `peek`

Proposed usage:

```powershell
tk show src/main.rs:120 -C 8
tk peek README.md:40
tk show src/main.rs
```

Parsing contract:

- Accept `FILE`.
- Accept `FILE:LINE` where `LINE` is a positive integer.
- Accept `FILE:LINE:COLUMN` only if column support is implemented; otherwise
  reject with a clear error.
- On Windows, do not treat a drive letter such as `C:\repo\file.rs` as a line
  delimiter. Only split a trailing `:<digits>` segment.

Text output contract:

- If a line is supplied, print `line - context` through `line + context`.
- Default context is `5`.
- Include line numbers.
- Mark or color the target line when color is enabled.
- If no line is supplied, behave as a concise file preview.

JSON output contract:

```json
{
  "path": "src/main.rs",
  "line": 120,
  "context": 8,
  "start_line": 112,
  "end_line": 128,
  "lines": [
    { "line": 120, "text": "...", "target": true }
  ]
}
```

MCP contract:

- Do not expose `show` in MCP until the CLI JSON output is stable.
- Once stable, expose `show` as a read-only MCP tool using canonical name only.

### Phase 5: Repo Orientation Workflow

Add one command for a fast codebase overview.

Preferred command names:

- canonical: `scan`
- alias: `overview`

Proposed usage:

```powershell
tk scan
tk scan src -L 2 -n 10
tk scan -j
```

Text output should include:

- target path
- shallow tree
- total files/dirs/bytes
- largest files
- recent files

JSON output should be a structured object:

```json
{
  "path": ".",
  "tree": { "name": ".", "type": "dir", "children": [] },
  "stats": { "files": 0, "dirs": 0, "bytes": 0 },
  "largest": [],
  "recent": []
}
```

Implementation should prefer shared helpers over shelling out to `tk` internally
unless that keeps the first slice dramatically simpler. If internal helpers do
not exist yet, create narrow data-returning helpers behind existing command
renderers.

MCP contract:

- Do not expose `scan` in MCP until text and JSON output are stable.
- Once stable, expose `scan` as read-only if the output is compact enough for
  typical MCP clients.

### Phase 6: Shell Completions

Add completions only after the alias contract stabilizes.

Proposed usage:

```powershell
tk completions powershell
tk completions bash
tk completions zsh
tk completions fish
```

This likely requires `clap_complete`. It should be its own implementation slice
because it adds a dependency and generated-output tests.

## API Or Data Contract

### Compatibility Contract

- Existing commands and flags remain valid.
- New aliases must map to existing command behavior unless explicitly specified
  as a new command.
- JSON output shape for existing commands must not change as part of the alias
  slice.
- MCP tool names remain canonical; human aliases do not automatically become
  MCP tools.

### Error Contract

- Command-level validation should continue to route through the global error
  handler so `--format json` can produce JSON errors.
- Clap parse-time errors are acceptable for pure syntax conflicts such as
  `--json` with `--format text`, but tests should document the behavior.

### Help Contract

- Root help should teach common workflows and visible aliases.
- Tests should assert key phrases and aliases rather than snapshotting the full
  help output.

## Rollout Plan

1. Implement Phase 1 aliases and short flags.
2. Add focused CLI tests for aliases, `-j`, `--json`, `--no-color`, `search -n`,
   `sort -k`, `stats -e`, and `clip --fallback`.
3. Update README and root help with the daily-driver path.
4. Implement Phase 2 `info PATH`.
5. Write proof for Phases 1-2.
6. Implement `show`/`peek` in a separate slice with JSON output and tests.
7. Implement `scan`/`overview` in a separate slice with JSON output and tests.
8. Consider shell completions after aliases are stable.

## Acceptance Criteria

Phase 1-2 are accepted when:

- `tk --help` shows the key shortcut layer or common recipes.
- Existing aliases still work: `l`, `lt`, `grep`, `fd`, and `find`.
- New aliases work: `s`, `rg`, `f`, `wc`, `hash`, `sum`, `view`, `big`, `new`,
  `fe`, `fn`, and `t`.
- `tk stats -e -j` emits valid JSON and matches `tk stats -t --format json`.
- `tk search PATTERN PATH -n` includes line numbers.
- `tk --no-color ls PATH` matches `tk --color never ls PATH` for ANSI behavior.
- `tk info PATH` matches `tk info -f PATH`.
- README documents the daily-driver path and all public aliases.
- MCP tool list remains canonical unless a later slice explicitly changes it.

Phase 4 is accepted when:

- `tk show FILE:LINE -C N` prints bounded context with line numbers.
- `tk show FILE:LINE -C N -j` emits the specified JSON shape.
- Windows-style paths are not misparsed as line targets.

Phase 5 is accepted when:

- `tk scan` gives a useful text overview without requiring multiple commands.
- `tk scan -j` emits the specified JSON object.
- The command is covered by integration tests and does not descend into `.git`.

## Rollback

- Alias and flag additions can be rolled back by reverting `src/main.rs`,
  README, and tests for the affected slice.
- `show` and `scan` should be introduced as standalone command variants so they
  can be reverted without changing existing command behavior.
- MCP exposure for any new command should be a separate patch so CLI rollback
  and MCP rollback are independent.

## Open Questions

- Should `--json` conflicting with `--format text` be a clap parse error or a
  command-level error that can respect JSON error formatting?
- Should `show` support `FILE:LINE:COLUMN`, or reserve column support for a
  later editor-integration spec?
- Should `scan` eventually include git metadata, such as branch status and
  untracked counts, or stay filesystem-only?
